Issue: https://issu.ee/492294149

**Vulnerability Title**
MiniTable size alignment overflow leads to zero-sized allocation and heap corruption

**CWE**
CWE-190: Integer Overflow or Wraparound  
CWE-131: Incorrect Calculation of Buffer Size

**Vulnerability Description**
The MiniTable builder computes a message layout size (`base`) and bounds-checks it against `UINT16_MAX`, but then aligns the 16-bit size up to `kUpb_Message_Align` without re-checking. When `base` is in the range 65529..65535 and not 8-byte aligned, `UPB_ALIGN_UP(base, 8)` produces 65536, which truncates to 0 when stored in the `uint16_t` `upb_MiniTable::size`. The resulting `MiniTable` reports `size == 0` while field offsets remain non-zero, so `_upb_Message_New()` allocates a zero-sized arena object and later field writes overflow into adjacent arena allocations. On non-ASAN builds, `upb_Arena_Malloc(0)` does not advance the arena pointer, which causes two message allocations to alias, turning a single field write into cross-object memory corruption.

Concrete boundary example: a message with 15,884 optional `int32` fields produces `base == 65532` (not 8-aligned). The alignment step rounds to 65536, truncates to 0, and then any field write (e.g., offset 1996 for `f1`) becomes out-of-bounds relative to a zero-sized allocation.

Affected code locations:
- `upb/mini_descriptor/decode.c:600` (size bound check against `UINT16_MAX`)
- `upb/mini_descriptor/decode.c:605` (stores `base` into 16-bit `size`)
- `upb/mini_descriptor/decode.c:684` (alignment step on 16-bit `size`)
- `upb/mini_table/internal/message.h:57` (layout size contract comment)
- `upb/mini_table/internal/message.h:59` (`uint16_t size` definition)
- `upb/message/internal/message.h:150` (uses `size` for allocation)
- `upb/message/internal/message.h:153` (`UPB_ASSUME` aligned size)
- `upb/message/internal/message.h:154` (`upb_Arena_Malloc(a, size)`)
- `upb/mem/internal/arena.h:72` (arena malloc fast path)
- `upb/mem/internal/arena.h:83` (pointer advance; span is zero when size is zero in non-ASAN builds)

This issue is distinct from the three known bugs you listed (DescriptorPool type confusion, `_upb_Array_Realloc` size_t overflow, `_upb_NoLocaleStrtod` stack overflow).

**Attack Preconditions**
- The target uses the upb runtime and builds MiniTables from untrusted or attacker-influenced schemas (for example, dynamic `FileDescriptorProto` input or MiniDescriptor blobs).
- The attacker can supply a sufficiently large message schema to place the layout size near the 65529..65535 boundary window (may vary slightly across 64-bit builds; for example, ~15k optional `int32` fields).
- The process allocates multiple messages from the same arena, so the zero-sized allocation can alias subsequent allocations.

#include <string.h>

#include "upb/port/def.inc"

int main(void) {
  const int N = 15884;
  size_t md_len = (size_t)N + 1;
  char *md = (char*)malloc(md_len);
  if (!md) {
    fprintf(stderr, "malloc failed\n");
    return 1;
  }
  md[0] = '$';            // kUpb_EncodedVersion_MessageV1
  memset(md + 1, '(', N); // kUpb_EncodedType_Int32

  upb_Arena* arena = upb_Arena_New();
  upb_Status status;
  upb_Status_Clear(&status);

  upb_MiniTable* mt = upb_MiniTable_Build(md, md_len, arena, &status);
  free(md);

  if (!mt) {
    fprintf(stderr, "MiniTable build failed: %s\n",
            upb_Status_ErrorMessage(&status));
    return 1;
  }

  printf("MiniTable size = %u\n", (unsigned)mt->UPB_PRIVATE(size));

  upb_Message* m1 = upb_Message_New(mt, arena);
  upb_Message* m2 = upb_Message_New(mt, arena);
  printf("m1=%p m2=%p (same? %s)\n", (void*)m1, (void*)m2,
         (m1 == m2 ? "YES" : "NO"));

  const upb_MiniTableField* f0 = upb_MiniTable_GetFieldByIndex(mt, 0);
  printf("f0.offset=%u\n", (unsigned)f0->UPB_ONLYBITS(offset));

  upb_Message_SetInt32(m1, f0, 123, arena);
  int32_t v = upb_Message_GetInt32(m2, f0, 0);
  printf("m2.f1=%d\n", v);

  return 0;
}

#include "upb/port/undef.inc"

**Impact Analysis**
Who can exploit: Any attacker who can feed an untrusted schema into a service that dynamically builds upb MiniTables (or accepts serialized MiniDescriptor/`FileDescriptorProto` input) and then parses or mutates messages allocated from a shared arena.

Security impact: The overflow forces `MiniTable::size` to 0 for a non-empty message, so `upb_Message_New()` returns a zero-sized allocation. Field writes then occur at non-zero offsets, corrupting adjacent arena objects. This yields a practical heap corruption primitive (cross-object write/aliasing) with attacker-controlled 32-bit values. In services that co-locate other objects in the same arena (arrays, maps, extensions, unknown fields), this can corrupt pointers and metadata, leading to crashes or potentially more severe impacts.
